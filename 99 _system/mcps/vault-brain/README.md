# vault-brain MCP

> Thin retrieval wrapper for the Mavis vault. Esalen posture: deterministic I/O + heuristic retrieval, M3 does the synthesis. Per [[03 Projects/Mavis-Apex-Architecture/03 The Custom MCP Arsenal]] design (MCP #1).

## What this is

A thin candidate generator. You give it a query; it returns the top-K most relevant notes from the vault, packed **anchor-ends style** (highest-signal at the start and end of the prompt, where M3's MSA sparse attention is strongest). M3 then reads the returned notes and synthesizes the actual answer.

The code is the deterministic I/O layer. The model is the synthesis layer. Per the [[99 _system/ESALEN-NOT-FOXCONN]] audit:

- ✅ We do NOT test things the model would have handled (semantic ranking is M3's job at call time)
- ✅ The code is a thin deterministic layer (BM25-ish scoring + heuristics, no LLM calls)
- ✅ The system trusts the model to finish (returns candidates, M3 does the final synthesis)

## What this is NOT

- Not a vector database. Embeddings are not stored on disk. The retrieval is keyword + tag + wikilink-graph + recency, scored and packed.
- Not a search engine. There is no full-text index, no Lucene, no Elasticsearch. The "index" is a single JSON file with metadata.
- Not a question-answering system. The model does the synthesis. vault-brain returns the context; M3 writes the answer.
- Not skillified yet. This is the raw MCP. The skill pack (markdown skill, integration test, resolver) comes after the test phase produces a stable shape.

## Usage

### Build the index (one-time, then on-demand)

```bash
cd 99 _system/mcps/vault-brain
python3 vault_brain.py --reindex
# Indexed 2541 notes → /Users/.../99 _system/mcps/vault-brain/index.json
```

Re-run when the vault has changed meaningfully (50+ new notes, or after a project completes).

### Run a one-shot search (the test query)

```bash
python3 vault_brain.py --query "What are all the operating postures and constraints we locked in today?" --top-k 10 --max-tokens 30000
```

Output: JSON with the top-10 relevant notes, anchor-ends arranged, with backlinks. The notes are rendered as markdown chunks ready to paste into an M3 context window.

### Run as MCP server

```bash
python3 vault_brain.py --serve
# Registers the "search" tool. Connect via stdio from any MCP client.
```

The MCP tool surface (per the Arsenal design):

```typescript
vault_brain.search({
  query: string,             // semantic search query
  top_k?: number,            // default 20
  max_total_tokens?: number, // default 200_000; respects M3's 1M budget
  include_backlinks?: boolean // default true
}) → {
  results: Array<{
    path: string,
    title: string,
    score: number,           // 0-1 (well, 0-N, the raw score)
    content: string,         // full note body (excerpt, actually 2000 chars)
    backlinks: string[],     // titles of notes that link here
    tags: string[]
  }>,
  total_tokens: number
}
```

(Note: the v0.1.0 implementation returns a richer object with `rendered` (markdown chunk), `signals` (which fields matched), and `tokens_estimate` per chunk. The Arsenal-spec fields are all present; we just expose more for debugging.)

## Scoring formula

For each note, the score is:

```
score = 10 * (title matches)
      + 5  * (tag matches)
      + 1 + 0.5 * sqrt(count) * (body matches, with log saturation)
      + 1  * (modified in last 7 days)
      + 0.5 * (modified in last 30 days)
```

Where "matches" means "at least one query token appears in this field."

The top-K notes by score are packed within the token budget using the **anchor-ends strategy**:

```
[1st, 3rd, 5th, ..., 4th, 2nd]
```

This puts the highest-scored note at position 0 (strong attention) and the second-highest at position N-1 (also strong attention). The middle gets the remaining notes. The pattern matches where M3's MSA is sharpest.

## Cost

- **Indexing**: ~5-10 seconds for a 2K-note vault. One-time + on-demand.
- **Storage**: ~5-10 MB for the index JSON.
- **Query**: ~50-100ms. No LLM call in the search path. M3 does the synthesis over the returned notes (a separate M3 inference call, not in this server).
- **The $100/month M3 inference budget** is consumed by the *consuming* M3 call, not by vault-brain itself.
