**Why this matters:** When three independent fields converge on the same principle, you have either a deep truth or a survivorship bias. The Esalen check: do the three domains agree on the *mechanism*, or only on the *result*? They agree on the mechanism.

## Connection 2 — TYPE B: "Routing is flow-reinforcement, in all three fields"

**The bridge:** In all three domains, the system **does not pre-compute the optimal routing.** It reinforces the routing that *is*, observes the consequences, and prunes what doesn't work. Mycelia thicken tubes that carry flow. Topological qubits are protected because the energy gap makes the wrong path energetically expensive. Generative codes unfold because each step *enhances* the whole — the wrong step is visible and repairable.

**Why this matters:** This is the *operational* principle behind all three. The Omni-Operator's Resolver should work the same way. Static ranking is the wrong default.

## Connection 3 — TYPE B: "Topology protects the essential structure; surface can change"

**The bridge:** Topological qubits are protected because their *information* is in the global state (the parity, the braiding history), not in the local one (the position of any one electron, the speed of one gate operation). Similarly, in the Wood Wide Web, the *network topology* persists even as individual hyphae grow and retract. And in generative codes, the *essential structure* (the values, the patterns, the sequence) persists even as the implementation changes.

**Why this matters:** This is the right model for vault persistence. The *content* of a note can change; the *form* (frontmatter, wikilinks, section structure) should be invariant. This is the [[Topological Memory Protection]] principle.

## Connection 5 — TYPE C: "The agent primitive is an anyon, not a function"

**The bridge:** A function f: A → B is a map. An anyon is a *transformation that depends on what other anyons have done to it.* The second is a richer model, and it's what we observe in agent systems. Agents don't just transform inputs; their *state* depends on the history of compositions.

**Why this matters:** The current vocabulary for agents (skill, pipeline, chain, graph) is the function-composition model. It cannot express: "the joint state of A, B, C is the only thing that matters for the next composition step." The anyon model can. The anyon model also gives us: non-commutativity (A∘B ≠ B∘A), universal gate sets (some compositions are universal), and topological error correction (a wrong braid is a correction error, not a value error).

## Connection 6 — TYPE D: "Workflows should be generative codes, not scripts"

**The bridge:** Alexander's evolution: patterns (1977) → sequences (2002) → generative codes (2004). Each step added a layer. Patterns are problem+solution. Sequences are patterns in order. Generative codes are sequences + the social/legal/procedural interactions that produce the structure in the real world.

A Mavis workflow is currently a *script* — a sequence of tool calls. The hypothesis is that a workflow should be a *generative code* — a sequence + the values, stakeholders, and boundary conditions. The script is the *output* of the generative code, not the workflow itself.

**Why this matters:** A new Mavis reading a script can execute it but doesn't *inherit* the values. A new Mavis reading a generative code understands the why. The generative code is the *membrane* between tool and culture. Without it, workflows are brittle; with it, workflows are alive.
